C++和GO属于强类型静态语言吗？

首先，C++和GO都是静态语言，所谓静态，即类型检查在编译时进行，而不是在运行时。这与静态编译、动态编译里的静态的概念具有一致性。GO是一种典型的强类型静态语言。变量必须严格符合其声明的类型，并且类型的转换需要显式地进行。
C++的说法众说纷纭，对于C++，通常我们认为它是强类型的，因为它需要显示声明变量类型。但是C++在某些情况下又具有弱类型的用法。例如，以下程序并不会error，似乎顺利的完成了隐式类型转换，将char上升为int，所以其在某些分类被归为弱类型。

Go语言相比C++在并发编程方面的优势

1.Go语言提供了用于并发编程的简单且易于使用的原语，例如goroutine和channel。
2.Go语言的运行时系统具有一个高效的调度器，可以自动将goroutine调度到可用的线程上执行。
3.Go语言具有自动垃圾回收机制，可以自动回收不再使用的内存，避免了手动管理内存的复杂性和错误。
4.Go语言通过提供channel和select语句等特性，支持了基于消息传递的并发模型和基于事件驱动的并发模型。

C++ Part

C++值传递和引用传递的区别？

1.值传递：形参是实参的拷贝，改变形参的值并不会影响外部实参的值。当函数内部需要修改参数，并且不希望这个改变影响调用者时，采用值传递。
2.引用传递：形参相当于是实参的“别名”，对形参的操作其实就是对实参的操作。

A:当参数占用内存较大时，值传递拷贝实参会产生多余的内存开销。在限制空间时，需要格外注意。当发现某段内存莫名占用预期的二倍，可能就是引用传递写成值传递了。

C++堆栈内存分配的区别？

1.栈：编译器分配释放，无需手动，向下生长。
2.堆：手动new\delete，向上生长。
A:使用new delete的时候需要慎重，在new和delete之间，如果程序异常中止，堆内存可能无法及时释放，引发内存泄露。
A:内存泄露比较常见就是堆内存泄露，指程序运行中根据需要分配通过malloc,new等从堆中分配的内存，完成后必须通过调用对应free,delete删除。

new和malloc的区别？

1.new分配失败报异常，malloc返回NULL。
2.new分配无需指定大小，malloc需要。
3.new,delete是运算符，可以重载，malloc,free是标准库函数，不可以重载。
4.new除了分配和释放内存外还会调用构造和析构函数。
5.new是类型安全的，malloc不是。

静态编译、动态编译、静态链接、动态链接？

1.静态链接：链接器将库添加到可执行程序中。
2.动态链接：运行时，完成链接过程。

Q:Symbol Undefined符号未定义错误？
A:当链接器须要对某个符号的引用进行重定位时，它就要确定这个符号的目标地址。链接器查找由所有输入目标文件的符号表组成的全局符号表，找到相应的符号后进行重定位。在链接器扫描完所有的输入目标文件之后，理论上所有这些未定义的符号都应该能够在全局符号表中找到，否则链接器就报符号未定义错误。
A:C++使用模板时，模板函数要被实例化后才能成为真正的函数，如果头文件中只有声明，没有定义，那编译器无法实例化该模板，最终导致链接错误。一般在模板定义文件的末尾，重新声明具现的函数模板（类模板）类型。可能出现Symbol Undefined的问题。

C++智能指针

C++智能指针主要是为了负责自动释放所指向的对象。
1.手写Shared_ptr

std::shared_ptr:它是一种共享拥有（shared ownership）的智能指针。多个 std::shared_ptr 对象可以共同拥有同一个资源，并且会自动地跟踪资源的引用计数。
2.std::unique_ptr：它是一种独占拥有（exclusive ownership）的智能指针。一个 std::unique_ptr 对象拥有资源的独占权，并且不能与其他 std::unique_ptr 或 std::shared_ptr 共享同一个资源。当 std::unique_ptr 对象超出作用域或被显式地释放时，它所拥有的资源会被释放。
3.std::weak_ptr:用于解决 std::shared_ptr 的循环引用问题。std::weak_ptr 允许你观测一个资源，而不会增加该资源的引用计数。std::weak_ptr 常用于解决 std::shared_ptr 的循环引用问题。当两个或多个对象相互持有 std::shared_ptr，形成循环引用时，资源可能无法被释放。通过将其中一个或多个 std::shared_ptr 改为 std::weak_ptr，可以打破循环引用，允许资源在不再被引用时被释放。

static

实现多个对象之间的数据共享。
1.静态变量：在函数内部使用 static 关键字声明的变量称为静态变量。静态变量在函数调用时不会被销毁，而是在整个程序的生命周期内保持其值。

2.静态成员变量
在类中使用 static 关键字声明的变量称为静态成员变量。静态成员变量属于类本身，而不是类的实例。所有类的实例共享同一个静态成员变量，且可以在不创建类的实例的情况下访问它。

3.静态成员函数
在类中使用 static 关键字声明的函数称为静态成员函数。静态成员函数不依赖于类的实例，可以直接通过类名调用。静态成员函数无法访问非静态成员变量，只能访问静态成员变量和其他静态成员函数。

C++纯虚函数和虚函数的区别

1.虚函数在基类中有一个默认的实现，但可以在派生类中进行重写。而纯虚函数在基类中没有默认的实现，它只是一个接口声明，要求派生类必须提供实现。
2.在基类中声明虚函数时，使用关键字 virtual 来标识。而在基类中声明纯虚函数时，需要在函数声明后面添加=0。
3.如果一个类中包含至少一个纯虚函数，那么该类被称为抽象类。抽象类不能被实例化，只能用作其他类的基类。而虚函数不会导致类成为抽象类。
PS:纯虚基类是一个包含至少一个纯虚函数的抽象类。
PS:虚基类是指在多重继承中使用的一种机制,与纯虚基类没有关系。当一个类同时继承自多个具有一个或多个公共基类的类时，可以将这些公共基类声明为虚基类，以避免派生类中出现多个实例。

Go的垃圾回收机制

Go语言使用了一种称为”三色标记法”的垃圾回收。
三色标记法基于追踪（tracing）的垃圾回收算法。它将所有的对象分为三种不同的颜色：白色、灰色和黑色。
1.白色：表示对象尚未被扫描。
2.灰色：表示对象已经被扫描，但是其引用的对象尚未被扫描。
3.黑色：表示对象已经被扫描，且其引用的对象也已经被扫描。
步骤：
1.初始阶段所有对象都是白色;
2.从root根出发扫描根对象,将它们引用到的对象都标记为灰色,其中root区域主要是当前程序运行到的栈和全局数据;
区域,是实时使用到的内存
3.将灰色对象标记为黑色,分析该灰色对象是否引用了其他对象,如果有引用其他对象,就将引用的对象标记为灰色;
4.不断分析灰色对象,直到灰色对象队列为空,此时白色对象即为垃圾,进行回收。

数据库

Mysql

MySQL事务特性

ACID:原子性、一致性、隔离性、持久性
原子性：事务必须视为一个不可分割的最小工作单元，整个事务中的所有操作要么全部提交成功，要么全部失败回滚，对于一个事务来说，不可能只执行其中的一部分操作，这就是事务的原子性。
一致性：数据库总是从一个一致性的状态转换到另一个一致性的状态。
隔离性：一个事务所做的修改在最终提交以前，对其他事务是不可见的。
持久性：一旦事务提交，则其所做的修改就会永久保存到数据库中。此时即使系统崩溃，修改的数据也不会丢失。

MySQL的事务隔离级别

读未提交：指一个事务还没提交时，它做的变更就能被其他事务看到。
读提交：指一个事务提交之后，它做的变更才能被其他事务看到。
可重复读：指一个事务执行过程中看到的数据，一直跟这个事务启动时看到的数据是一致的，MySQL InnoDB 引擎的默认隔离级别。
串行化：会对记录加上读写锁，在多个事务对这条记录进行读写操作时，如果发生了读写冲突的时候，后访问的事务必须等前一个事务执行完成，才能继续执行。

MySQL索引底层结构

B+Tree,叶子节点存放数据，非叶子节点只存放索引，每个节点里的数据是按照主键的顺序存放的。在叶子节点中，包含了所有索引值信息，并且每一个叶子节点都指向下一个叶子节点，形成一个链表。B+Tree 存储千万级的数据只需要 3-4 层高度就可以满足，千万级的表查询目标数据最多需要 3-4 次磁盘 I/O。

B+Tree对比BTree: BTree只在叶子节点存放数据，而B树的非叶子节点也要存放数据，所以B+Tree的单个节点数据量更小，在相同的磁盘I/O次数下，能够查询更多的节点。B+Tree的叶子节点采用的是双链表连接，适合MySQL中常见的基于范围的顺序查找。

B+Tree对比二叉树：对于有N个叶子节点的B+Tree，其搜索复杂度为O(logdN)，其中d表示节点允许的最大子节点个数。在实际的应用当中，d的值是大于100的，即使数据达到千万级别，B+Tree的高度依然维持在34层左右，一次数据查询只需要34次磁盘I/O就可以查到。二叉树的搜索复杂度为O(logN)，I/O次数要更多。

B+Tree对比Hash：Hash在做等值查询时效率高，搜索复杂度为O(1)，但是Hash表不适合做范围查询。

2.唯一索引：索引列的值必须唯一，可以为null
3.主键索引：值必须唯一且不能为null

最左匹配原则

使用联合索引时，按照最左优先的方式进行索引的匹配。
因为(a, b, c)联合索引，是先按 a 排序，在 a 相同的情况再按 b 排序，在 b 相同的情况再按 c 排序。所以，b 和 c 是全局无序，局部相对有序的，这样在没有遵循最左匹配原则的情况下，是无法利用到索引的。利用索引的前提是索引里的 key 是有序的。

使用索引的缺陷

索引提高了查询速度，降低了表的更新速度，索引需要占用空间。

悲观锁和乐观锁

悲观锁是一种较保守的策略，它假设在并发环境中会发生冲突，因此在访问数据之前会先获取锁，确保其他事务无法同时修改数据。在悲观锁的模式下，当一个事务要访问数据时，它会先对数据进行加锁，其他事务必须等待锁的释放才能继续访问或修改数据。

乐观锁则采取了相反的策略，它假设并发冲突的概率较低，并且认为事务之间的冲突是不常见的。在乐观锁的模式下，当一个事务要修改数据时，它不会立即加锁，而是先读取数据并记录下版本号或时间戳。当事务提交修改时，会检查数据的版本号或时间戳是否与读取时一致。如果一致，则说明期间没有其他事务修改过数据，可以提交修改；如果不一致，则说明有其他事务修改过数据，事务需要回滚并重新处理。

MVCC多版本并发控制

MVCC是数据库管理系统中使用的一种技术，用于处理多个事务对数据的并发访问。可以做到即使有读写冲突时，也能够做到不加锁，非阻塞并发读。
数据库中存在多个版本的数据，某个事务对其进行操作的时候，需要判断读取哪个版本的数据。可以高效避免脏写（修改丢失）、脏读、不可重复读、幻读的问题。
1.脏写（Dirty Write）是指一个事务对数据进行了修改，但是由于某种原因（如事务失败或回滚），该修改未能提交，导致数据的一致性受损。
2.脏读（Dirty Read）是指一个事务读取了另一个事务尚未提交的数据。如果这个事务最终回滚，那么读取的数据就是无效的，从而导致脏读。
3.不可重复读（Non-repeatable Read）是指一个事务在相同的查询条件下多次读取同一数据，但在读取过程中，其他事务对该数据进行了修改或删除，导致多次读取的结果不一致。
4.幻读（Phantom Read）是指一个事务在相同的查询条件下多次执行同一查询，但在执行过程中，其他事务插入或删除了满足查询条件的数据，导致多次查询的结果不一致。

慢查询的原因

1.索引不足：如果查询的表没有合适的索引，MySQL需要遍历整个表才能找到匹配的记录，这会导致查询变慢。可以通过添加索引来优化查询性能。
2.数据库设计问题：如果数据库设计不合理，例如表过于庞大、列过多等，查询时可能需要耗费大量时间。这时可以通过优化数据库设计来解决问题。
3.数据库服务器负载过高：如果MySQL服务器上同时运行了太多的查询，会导致服务器负载过高，从而导致查询变慢。
4.查询语句复杂：复杂的查询语句可能需要耗费更多的时间才能完成。

MySQL的三种日志

1.undo log(回滚日志)：在事务没提交之前，Innodb会先记录更新前的数据到undo log中，回滚时利用 undo log 来进行回滚。
2.redo log(重做日志)：redo log是物理日志，记录了某个数据页做了什么修改，每当执行一个事务就会产生一条或者多条物理日志。
3.binlog(归档日志)：在完成一条更新操作后，Server 层会生成一条 binlog，等之后事务提交的时候，会将该事物执行过程中产生的所有 binlog 统一写入 binlog 文件。binlog 文件是记录了所有数据库表结构变更和表数据修改的日志，不会记录查询类的操作。

Redis

Redis为什么快

1.基于内存操作
2.高效的数据结构
3.采用单线程
4.I/O多路复用

为什么采用redis作为MySQL的缓存

MySQL是数据库系统，对于数据的操作需要访问磁盘，而将数据放在Redis中，需要访问就可以直接从内存中获取，避免磁盘I/O，提高操作的速度。
使用Redis+MySQL结合的方法可以有效的提高系统的QPS。

如何保证数据库和缓存的一致性

1.Cache Aside
先从缓存中读取数据，如果没有就再去数据库里面读数据，然后把数据放回缓存中，如果缓存中可以找到数据就直接返回数据；更新数据的时候先把数据持久化到数据库，然后再让缓存失效。
2.Read/Write Through
把更新数据库（Repository）的操作由缓存代理，应用认为后端是一个单一的存储，而存储自己维护自己的缓存。
3.Write Behind
在更新数据的时候，只更新缓存，不更新数据库，而缓存会异步地批量更新数据库。

Python - sqlParse

sqlparse是用于Python的非验证SQL解析器。它支持解析、拆分和格式化SQL语句。
sqlParse解析SQL语句，产生查询树和字典。
字符流->词法分析器->Token流->语法分析器->AstTree->树分析器
PS:AstTree抽象语法树，可用于分析和处理源代码，也可以用于重构等。

设计模式

单例模式

保证一个类仅有一个实例，并提供一个访问它的全局访问点。

工厂模式

定义一个创建对象的接口，但由子类决定需要实例化哪一个类。

观察者模式

定义对象的一种一对多的关系，当一个对象的状态发生改变时，所有依赖于它的对象都能够得到通知并自动更新。

装饰模式

动态的给一个对象添加一些额外的职责。提供了用子类扩展功能的一个灵活替代。
