数据库

基础

基本概念

1.元组：关系是一张表，表中的每行（即数据库中的每条记录）就是一个元组，每列就是一个属性。在二维表里，元组也称为行。
2.码：码是唯一标识实体的属性，对应表中的列。
3.候选码：若关系中的某一属性或属性组的值能够唯一标识一个元组，而其任何子集都不能再标识，则称该属性组为候选码。
4.主码：主码从候选码中选出。一个实体集中只能有一个主码。
5.外码：如果一个关系中的一个属性是另外一个关系中的主码则这个属性为外码。
6.主属性：候选码中出现过的属性称为主属性。
7.非主属性：不包含在任何一个候选码中的属性称为非主属性。

8.函数依赖：若在一张表中，在属性X的值确认的情况下，必定能确定属性Y的值，那么就可以说Y函数依赖于X，写作X->Y。
9.部分函数依赖：如果X->Y,并且存在X的真子集X0，使得 X0→Y，则称 Y 对 X 部分函数依赖。
10.在一个关系中，若某个非主属性数据项依赖于全部关键字称之为完全函数依赖。
11.传递函数依赖：在关系 R(学号 , 姓名, 系名，系主任)中，学号 → 系名，系名 → 系主任，所以存在非主属性系主任对于学号的传递函数依赖。

数据库范式

1NF第一范式：属性不可再分。
2NF第二范式：在1NF的基础上，消除非主属性对于码的部分函数依赖。
3NF第三范式：3NF 在 2NF 的基础之上，消除了非主属性对于码的传递函数依赖。

6.数据库运行与维护

NoSQL 非关系型数据库

主要针对的是键值、文档以及图形类型数据存储。
Q:什么是关系型数据库？
A:关系型数据库是一种建立在关系模型的基础上的数据库。

SQL和NoSQL的区别

1.数据存储类型：SQL结构化存储，具有固定行和列；NoSQL非结构化存储。
2.目标：SQL减少数据重复；NoSQL提供可扩展性，减少大规模数据的存储成本。
3.举例：SQL(Oracle,MySql,PostgreSQL),NoSQL(MongoDb,Redis)。
4.ACID：SQL提供ACID属性，NoSQL不一定。
5.性能：SQL性能通常取决于磁盘子系统，需要优化查询、索引、表结构；NoSQL性能通常由底层硬件集群大小、网络延迟以及调用应用程序来决定。
6.扩展：SQL垂直扩展，使用性能更强的服务器、读写分离、分库分表等方法；NoSQL横向扩展，基于分片机制。

Mysql

MySQL基础

MySQL字符集

utf8,utf8mb4

MySQL的存储引擎

1.InnoDB 支持行级别的锁粒度，MyISAM 不支持，只支持表级别的锁粒度。
2.MyISAM不提供事务支持，InnoDB提供事务支持，实现了四个隔离级别。
3.MyISAM不支持外键，而InnoDB支持。
4.MyISAM不支持MVCC，而InnoDB支持。
5.虽然MyISAM引擎和InnoDB引擎都是使用B+Tree作为索引结构，但是二者的实现方式不太一样。
6.MyISAM不支持数据库崩溃后的安全回复，InnoDB支持。
7.InnoDB的性能比MyISAN更加强大。

MySQL事务特性

事务：逻辑上的一组操作，要么都执行，要么都不执行。
ACID:原子性、一致性、隔离性、持久性
原子性：事务必须视为一个不可分割的最小工作单元，整个事务中的所有操作要么全部提交成功，要么全部失败回滚，对于一个事务来说，不可能只执行其中的一部分操作，这就是事务的原子性。
一致性：数据库总是从一个一致性的状态转换到另一个一致性的状态。
隔离性：一个事务所做的修改在最终提交以前，对其他事务是不可见的。
持久性：一旦事务提交，则其所做的修改就会永久保存到数据库中。此时即使系统崩溃，修改的数据也不会丢失。

MySQL的事务隔离级别

读未提交：指一个事务还没提交时，它做的变更就能被其他事务看到。
读提交：指一个事务提交之后，它做的变更才能被其他事务看到。
可重复读：指一个事务执行过程中看到的数据，一直跟这个事务启动时看到的数据是一致的，MySQL InnoDB 引擎的默认隔离级别。
串行化：会对记录加上读写锁，在多个事务对这条记录进行读写操作时，如果发生了读写冲突的时候，后访问的事务必须等前一个事务执行完成，才能继续执行。

MySQL索引

MySQL索引底层结构

B+Tree,叶子节点存放数据，非叶子节点只存放索引，每个节点里的数据是按照主键的顺序存放的。在叶子节点中，包含了所有索引值信息，并且每一个叶子节点都指向下一个叶子节点，形成一个链表。B+Tree 存储千万级的数据只需要 3-4 层高度就可以满足，千万级的表查询目标数据最多需要 3-4 次磁盘 I/O。

B+Tree对比BTree: BTree只在叶子节点存放数据，而B树的非叶子节点也要存放数据，所以B+Tree的单个节点数据量更小，在相同的磁盘I/O次数下，能够查询更多的节点。B+Tree的叶子节点采用的是双链表连接，适合MySQL中常见的基于范围的顺序查找。

B+Tree对比二叉树：对于有N个叶子节点的B+Tree，其搜索复杂度为O(logdN)，其中d表示节点允许的最大子节点个数。在实际的应用当中，d的值是大于100的，即使数据达到千万级别，B+Tree的高度依然维持在34层左右，一次数据查询只需要34次磁盘I/O就可以查到。二叉树的搜索复杂度为O(logN)，I/O次数要更多。

B+Tree对比Hash：Hash在做等值查询时效率高，搜索复杂度为O(1)，但是Hash表不适合做范围查询。

MySQL索引种类

1.主键索引PrimaryKey：一张数据表只能有一个主键。
2.二级索引：二级索引的叶子节点存储的数据是主键。
(1)唯一索引：唯一索引的属性列不能出现重复的数据，但是允许数据为 NULL。 建立唯一索引的目的大部分时候都是为了该属性列的数据的唯一性，而不是为了查询效率。
(2)普通索引：普通索引的唯一作用就是为了快速查询数据。
(3)前缀索引：前缀索引只适用于字符串类型的数据。
(4)全文索引：为了检索大文本数据中的关键字的信息。
3.聚簇索引和非聚簇索引：
聚簇索引：索引结构和数据一起存放的索引。
非聚簇索引：索引结构和数据分开存放的索引。
4.覆盖索引：如果一个索引包含所有需要查询的字段的值，我们就称之为覆盖索引。
5.联合索引：使用表中的多个字段创建索引，就是联合索引。

最左匹配原则

使用联合索引时，按照最左优先的方式进行索引的匹配。
因为(a, b, c)联合索引，是先按 a 排序，在 a 相同的情况再按 b 排序，在 b 相同的情况再按 c 排序。所以，b 和 c 是全局无序，局部相对有序的，这样在没有遵循最左匹配原则的情况下，是无法利用到索引的。利用索引的前提是索引里的 key 是有序的。

使用索引的缺陷

索引提高了查询速度，降低了表的更新速度，索引需要占用空间。

如何正确使用索引

选择合适的字段创建索引
1.索引字段的数据应该尽量不为 NULL
2.创建索引的字段应该是查询操作非常频繁的字段
3.被作为 WHERE 条件查询的字段，应该被考虑建立索引
4.被经常频繁用于连接的字段，考虑建议索引。
5.被频繁更新的字段，不推荐建立索引。
6.建议单张表索引不超过5个。
7.避免索引失效。
8.删除长期未使用的索引。
9.字符串类型的字段使用前缀索引代替普通索引。

常见索引失效的情况

1.创建组合索引，但未遵循最左匹配原则。
2.在索引列上进行计算、函数、类型转换等操作。
3.以%开头的like查询
4.查询条件中使用OR，且OR前后条件中有一个列没有索引，涉及的索引都不会被使用到。
5.IN取值范围较大的时候。

MySQL锁

悲观锁和乐观锁

悲观锁是一种较保守的策略，它假设在并发环境中会发生冲突，因此在访问数据之前会先获取锁，确保其他事务无法同时修改数据。在悲观锁的模式下，当一个事务要访问数据时，它会先对数据进行加锁，其他事务必须等待锁的释放才能继续访问或修改数据。

乐观锁则采取了相反的策略，它假设并发冲突的概率较低，并且认为事务之间的冲突是不常见的。在乐观锁的模式下，当一个事务要修改数据时，它不会立即加锁，而是先读取数据并记录下版本号或时间戳。当事务提交修改时，会检查数据的版本号或时间戳是否与读取时一致。如果一致，则说明期间没有其他事务修改过数据，可以提交修改；如果不一致，则说明有其他事务修改过数据，事务需要回滚并重新处理。

MySQL表级锁和行级锁

1.表级锁：MySQL中锁定粒度最大的一种锁，对当前操作的整张表加锁，实现简单，高并发下效率低。针对非索引字段加的锁。
2.行级锁：MySQL中锁定粒度最小的一种锁，只针对当前操作的行记录进行加锁。加锁开销大，容易死锁。针对索引字段加的锁。
A：如果WHERE条件中字段没有命中唯一索引或者索引失效的话，行级锁会退化成表级锁，影响性能。

InnoDB有哪几类行锁

1.记录锁：单行上锁。
2.间隙锁：锁定一个范围，不包含记录本身。
3.临键锁：锁定一个范围，包含记录本身。

共享锁和排他锁

表级锁和行级锁都存在共享锁和排他锁这两类
共享锁S锁：读锁，事务在读取记录的时候获取共享锁，允许多个事务同时获取。
排他锁X锁：写锁，事务在修改记录的时候获取排他锁，不允许多个事务同时获取。
由于MVCC语句的存在，对于一般的SELECT语句，InnoDB不会加任何锁。

意向锁

判断是否可以堆某个表使用表锁

当前读和快照读

当前读：给行记录加X锁或S锁。
快照读：如果读取的记录正在执行UPDATE/DELETE操作，读取操作不会因此去等待记录上的X锁释放，而是去读取行的一个快照。

自增锁

自增主键会涉及一种比较特殊的表级锁。

MVCC多版本并发控制

MVCC是数据库管理系统中使用的一种技术，用于处理多个事务对数据的并发访问。可以做到即使有读写冲突时，也能够做到不加锁，非阻塞并发读。
数据库中存在多个版本的数据，某个事务对其进行操作的时候，需要判断读取哪个版本的数据。可以高效避免脏写（修改丢失）、脏读、不可重复读、幻读的问题。
1.脏写（Dirty Write）是指一个事务对数据进行了修改，但是由于某种原因（如事务失败或回滚），该修改未能提交，导致数据的一致性受损。
2.脏读（Dirty Read）是指一个事务读取了另一个事务尚未提交的数据。如果这个事务最终回滚，那么读取的数据就是无效的，从而导致脏读。
3.不可重复读（Non-repeatable Read）是指一个事务在相同的查询条件下多次读取同一数据，但在读取过程中，其他事务对该数据进行了修改或删除，导致多次读取的结果不一致。
4.幻读（Phantom Read）是指一个事务在相同的查询条件下多次执行同一查询，但在执行过程中，其他事务插入或删除了满足查询条件的数据，导致多次查询的结果不一致。

慢查询的原因

1.索引不足：如果查询的表没有合适的索引，MySQL需要遍历整个表才能找到匹配的记录，这会导致查询变慢。可以通过添加索引来优化查询性能。
2.数据库设计问题：如果数据库设计不合理，例如表过于庞大、列过多等，查询时可能需要耗费大量时间。这时可以通过优化数据库设计来解决问题。
3.数据库服务器负载过高：如果MySQL服务器上同时运行了太多的查询，会导致服务器负载过高，从而导致查询变慢。
4.查询语句复杂：复杂的查询语句可能需要耗费更多的时间才能完成。

MySQL日志

MySQL的三种日志

1.undo log(回滚日志)：在事务没提交之前，Innodb会先记录更新前的数据到undo log中，回滚时利用 undo log 来进行回滚。
2.redo log(重做日志)：redo log是物理日志，记录了某个数据页做了什么修改，每当执行一个事务就会产生一条或者多条物理日志。
3.binlog(归档日志)：在完成一条更新操作后，Server 层会生成一条 binlog，等之后事务提交的时候，会将该事物执行过程中产生的所有 binlog 统一写入 binlog 文件。binlog 文件是记录了所有数据库表结构变更和表数据修改的日志，不会记录查询类的操作。

redolog VS binlog

1.redo log是InnoDB引擎特有的，binlog是MySQL的Server层实现的，所有引擎都可以使用。
2.redo log是物理日志，记录对数据库进行的物理级别的修改操作。循环写入，当写满一轮后可以覆盖之前的日志，它记录了数据页的物理修改，而不关心具体的SQL语句；记录了对数据库进行的逻辑级别的修改操作，以追加方式写入，不会覆盖之前的日志，因此可以保留完整的操作历史。
3.redo log用于在崩溃恢复和故障恢复等情况下恢复数据库的一致性，binlog可以用于数据复制、数据恢复、数据备份和数据审计等用途。

MySQL高性能

索引下推

索引下推（Index Condition Pushdown） 是 MySQL 5.6 版本中提供的一项索引优化功能，可以在非聚簇索引遍历过程中，对索引中包含的字段先做判断，过滤掉不符合条件的记录，减少回表次数。

负载均衡

负载均衡 指的是将用户请求分摊到不同的服务器上处理，以提高系统整体的并发处理能力以及可靠性。一般使用Nginx来做负载均衡。
负载均衡的方法
1.随机法：最简单粗暴的负载均衡算法，所有的服务器被访问到的概率都是相同的，如果配置权重的话，权重越高的服务器被访问的概率就越大。
2.轮询法：轮询法是挨个轮询服务器处理，也可以设置权重。
3.两次随机法：两次随机法在随机法的基础上多增加了一次随机，多选出一个服务器。
4.哈希法：将请求的参数信息通过哈希函数转换成一个哈希值。
5.最快响应时间法

MySQL主从复制的原理

MySQL binlog(binary log 即二进制日志文件) 主要记录了MySQL数据库中数据的所有变化。

MySQL读写分离

读写分离主要是为了将对数据库的读写操作分散到不同的数据库节点上。小幅提升写性能，大幅提升读性能。

MySQL分库分表

遇到下面几种场景可以考虑分库分表：
1.单表的数据达到千万级别以上，数据库读写速度比较缓慢。
2.数据库中的数据占用的空间越来越大，备份时间越来越长。
3.应用的并发量太大

Redis

Redis为什么快

1.基于内存操作
2.高效的数据结构
3.采用单线程
4.I/O多路复用

为什么采用redis作为MySQL的缓存

MySQL是数据库系统，对于数据的操作需要访问磁盘，而将数据放在Redis中，需要访问就可以直接从内存中获取，避免磁盘I/O，提高操作的速度。
使用Redis+MySQL结合的方法可以有效的提高系统的QPS。

如何保证数据库和缓存的一致性

1.Cache Aside
先从缓存中读取数据，如果没有就再去数据库里面读数据，然后把数据放回缓存中，如果缓存中可以找到数据就直接返回数据；更新数据的时候先把数据持久化到数据库，然后再让缓存失效。
2.Read/Write Through
把更新数据库（Repository）的操作由缓存代理，应用认为后端是一个单一的存储，而存储自己维护自己的缓存。
3.Write Behind
在更新数据的时候，只更新缓存，不更新数据库，而缓存会异步地批量更新数据库。

Python - sqlParse

sqlparse是用于Python的非验证SQL解析器。它支持解析、拆分和格式化SQL语句。
sqlParse解析SQL语句，产生查询树和字典。
